Le language Bash

Pour programmer les challenges de manière autonome, on utilise le langage de référence du terminal Unix: bash. La version proposée ici est allégé et seul certaines fonctionnalités sont implémentées.
-variable: une variable contient de l'information sous forme de texte et elle permet notamment de faciliter l'écriture des scripts, la notation est $+le nom de la variable, dans le cas ou l'on souhaite faire des concaténations
on dispose d'une autre écriture : ${nom variable}. Enfin pour affecter une variable, on utilise l'écriture: "nom variable"="valeur de la variable". On note notamment qu'il ne doit pas y avoir d'escape entre ces différents
éléments. Une commande dont l'exécution s'est déroulée sans problème affecte à la variable $? la valeur 0 et 1 dans le cas contraire.
- création de fonctions: une fonction peut avoir n arguments et peut renvoyer un résultat. Dans le contexte local de la fonction, on accède aux arguments avec les variables: $1 ... $n et $# indique le nombre d'arguments (ici n).
On les définit de la manière suivante:
f(){ code }
Un appel de la fonction s'effectue en utilisant l'écriture: f
- L'interpréteur gére également les sous commandes de la forme ("commande"), ces dernières sont lancés avec un contexte local et leur résultat peut être utilisées dans l'entrée standard d'une commande ou pour définir une variable.
exemple:
- des blocks de contrôle: comme la plupart des langages, bash dispose de plusieurs block de contrôle, on en donne les syntaxes ici: if "condition" then
"code"
else
"code"
fi
Le else est optionnel.
while "condition" do
"code"
done
for k in "list" do
"code"
done
La liste correspond à une succession de chaînes de caractères: par exemple comme 1 2 3 4 5, for est exécuté ici 5 fois.
case "expression" in
i) "code" ;;
j) "code" ;;
esac
i,j représente des chaînes de caractères et peuvent intégrer des éléments de regex.
- On peut définir des alias qui permettent de simplifier l'écriture des commandes, On créé un alias de la manière suivante:
alias f="commande"
-Des opérateurs binaires sont disponibles pour effectuer des commandes complexes:
"commande1" | "commande2" redirige la sortie standard de commande1 avec l'entrée standard de commande2
"commande" > file : permet de rediriger le flux stdout vers un fichier, si le fichier existe déjà, à condition de possèder les permissions nécessaires, ce dernier est écrasé.
"commande" >> file : quasi identique à > mais le flux est ajouté en fin de fichier.
"commande1" || "commande2" : si la commande1 est un succès ($? =0), la commande 2 n'est pas exécuté. Si la commande1 est un échec ($? !=0), la commande 2 est exécuté.
"commande1" && "commande2": même principe que || mais la commande2 est exécuté uniquement si commande1 est un succès.
command < file : redirige un fichier dans l'entrée standard de la commande.
- Le système de fichier est similaire à Unix et permet la création de lien physique, symbolique ainsi que la gestion des permissions.
- En complément du point précedent, un système de groupes et d'utilisateur a été implémenté.
- Pour faciliter la compréhension du terminal, les codes couleurs et le système de changement de polices habituellement utilisés en Unix on également été ajouté par exemple:
echo "\e[31mCheval\e[0m" : affiche le mot Cheval en rouge.
- La commande music permet de charger des morceaux musicaux stockés dans le système de fichier.

squelette d'un challenge

Un challenge se décompose en plusieurs éléments: un système de fichier composés d'une racine uniquement accessibles aux administrateurs (personnes dans le groupe root, en général
root et shaper) avec plusieurs éléments:
commandScript.sh : ce script est lancé à chaque validation d'une commande par le challenger.
initScript.sh/tout les scripts dans le dossier scripts : ces scripts sont lancés après le chargement du challenge.
mainScript.sh : ce script est exécutable par le challenger.
endScript.sh : ce script se déclenche lors de la fin (victoire) d'un challenge.
